# ROADMAP проекта

## Этап 1. Базовый сбор данных с ВМ

### Цель

Создать устанавливаемое приложение, способное получить базовую информацию о ВМ и передать её в `node_exporter` через textfile collector.

### Состав работ

* [ ] Реализовать установку приложения на ВМ.
* [ ] Реализовать ручной запуск приложения.
* [ ] Реализовать определение дистрибутива и версии ОС на основании `/etc/os-release`.
* [ ] Сформировать метрику в формате Prometheus.
* [ ] Записать метрику в каталог textfile collector `node_exporter`.
* [ ] Обеспечить корректное обновление файла метрик.
* [ ] Добавить метрику успешности выполнения сбора.

### Критерий завершения

Этап завершён, если информация о текущей ОС ВМ доступна в выводе:

```text
http://localhost:9100/metrics
```

---

## Этап 2. Контроль базовой настройки ВМ

### Цель

Добавить проверку соответствия ВМ утверждённой базовой настройке.

### Состав работ

* [ ] Определить структуру YAML-конфигурации базовой настройки.
* [ ] Описать в конфигурации требования к ВМ.
* [ ] Реализовать чтение и проверку YAML-конфигурации.
* [ ] Реализовать проверки параметров базовой настройки.
* [ ] Сформировать итоговый статус соответствия ВМ базовой настройке.
* [ ] Передать результат проверки в `node_exporter`.
* [ ] Сохранить локально информацию о выявленных отклонениях.

### В состав базовой настройки могут входить

* обязательные пакеты;
* обязательные сервисы;
* настройки DNS;
* настройки синхронизации времени;
* настройки почтового релея;
* другие стандартные параметры подготовки ВМ.

### Критерий завершения

Этап завершён, если результат проверки базовой настройки ВМ доступен в выводе:

```text
http://localhost:9100/metrics
```

и по метрике можно определить:

* проверка выполнена успешно;
* ВМ соответствует базовой настройке;
* ВМ не соответствует базовой настройке;
* проверку выполнить не удалось.

## Этап 3. Первый профиль СПО: Nginx

### Цель

Добавить отдельный слой контроля системного программного обеспечения на примере Nginx.

### Состав работ

* [ ] Определить формат отдельного YAML-профиля СПО.
* [ ] Создать профиль для Nginx.
* [ ] Описать в профиле требования к установке и настройке Nginx.
* [ ] Реализовать проверку наличия Nginx.
* [ ] Реализовать сбор текущей версии Nginx.
* [ ] Реализовать проверку состояния сервиса.
* [ ] Реализовать проверку автозапуска сервиса.
* [ ] Реализовать проверку валидности конфигурации.
* [ ] Определить способ контроля изменений конфигурационных файлов.
* [ ] Передать итоговые результаты слоя СПО в `node_exporter`.

### Критерий завершения

Этап завершён, если приложение по профилю Nginx определяет:

* установлен ли Nginx;
* запущен ли сервис;
* включён ли автозапуск;
* валидна ли конфигурация;
* изменялись ли контролируемые конфигурационные файлы;
* какая версия Nginx установлена;

и соответствующие метрики доступны через:

```text
http://localhost:9100/metrics
```

## Этап 4. Масштабирование слоя СПО

### Цель

Поддержать все необходимые типы СПО и решить задачу корректного назначения профилей конкретным ВМ.

### Состав работ

* [ ] Определить перечень типов СПО, которые необходимо контролировать.
* [ ] Для каждого типа СПО определить набор проверок.
* [ ] Для каждого типа СПО создать отдельный профиль.
* [ ] Реализовать недостающие механизмы сбора и проверки.
* [ ] Исключить дублирование одинаковых проверок между профилями.
* [ ] Определить способ назначения ВМ только тех профилей СПО, которые к ней относятся.
* [ ] Определить способ доставки профилей на ВМ.
* [ ] Обработать ситуацию отсутствующего или ошибочно назначенного профиля.
* [ ] Проверить, что неподходящий профиль не создаёт ложный штатный результат.
* [ ] Проверить масштабирование решения на несколько групп ВМ.

### Ключевой проектный вопрос

Необходимо определить механизм, который гарантирует, что:

```text
ВМ с Nginx получает профиль Nginx;
ВМ со Sphinx получает профиль Sphinx;
ВМ с несколькими типами СПО получает соответствующий набор профилей.
```

Ошибочно назначенный профиль должен определяться как ошибка конфигурации, а не как обычное несоответствие СПО.

### Критерий завершения

Этап завершён, если:

* для каждого поддерживаемого типа СПО существует профиль;
* ВМ получает только применимые к ней профили;
* ошибочное назначение профиля выявляется;
* добавление нового типа СПО не требует переработки базовой логики приложения;
* результаты всех профилей доступны через `node_exporter`.

## Этап 5. Контроль репликации и актуальности данных

### Цель

Определить и реализовать контроль состояния синхронизации и отставания данных.

### Состав работ

* [ ] Составить перечень систем, в которых используется репликация или синхронизация.
* [ ] Определить штатные механизмы контроля для каждого типа системы.
* [ ] Определить, какие показатели уже собираются существующими exporters.
* [ ] Исключить повторный сбор уже доступных данных.
* [ ] Определить недостающие показатели.
* [ ] Решить, какие данные должны собираться локальным приложением.
* [ ] Решить, какие данные должны собираться другими специализированными средствами.
* [ ] Для систем без штатного механизма определить косвенную методику оценки.
* [ ] Реализовать сбор недостающих данных.
* [ ] Передать необходимые показатели в мониторинг.

### Основной принцип

Если система имеет штатные показатели состояния репликации, используются они.

Если штатных показателей нет, оценка может выполняться по:

* количеству объектов или записей;
* объёму данных;
* времени последнего изменения;
* времени последней успешной синхронизации;
* другим согласованным признакам.

### Критерий завершения

Этап завершён, если для каждого выбранного типа системы определены:

* источник данных;
* методика оценки;
* уже существующие метрики;
* недостающие метрики;
* способ их получения;
* формат передачи результата в мониторинг.

## Этап 6. Контроль версий ППО

### Цель

Добавить сбор версий прикладного программного обеспечения, включая приложения в Kubernetes.

### Состав работ

* [ ] Определить перечень контролируемого ППО.
* [ ] Определить источник текущей версии для каждого типа приложения.
* [ ] Реализовать получение версий приложений, установленных непосредственно на ВМ.
* [ ] Исследовать способы определения версий приложений в Kubernetes.
* [ ] Определить, что считается целевой версией приложения.
* [ ] Определить источник целевой версии.
* [ ] Разделить текущую и целевую версии.
* [ ] Определить обработку приложений с несколькими одновременно используемыми версиями.
* [ ] Определить формат передачи версии в мониторинг.
* [ ] Реализовать первые процедуры сбора версий ППО.

### Варианты источников текущей версии

* файл версии;
* RPM-пакет;
* manifest приложения;
* build information;
* Git commit;
* версия контейнерного образа;
* label или annotation Kubernetes;
* локальный API приложения.

### Открытые вопросы этапа

* где хранится целевая версия;
* кто её определяет;
* как связать приложение с его целевой версией;
* как обрабатывать разные версии в разных ЦОД;
* как определять версию приложения внутри Kubernetes;
* считать ли тег контейнерного образа достаточным источником версии.

### Критерий завершения

Этап завершён, если для выбранных типов ППО приложение или связанный с ним механизм:

* определяет текущую версию;
* получает или знает целевую версию;
* формирует данные для сравнения;
* передаёт информацию в мониторинг.

# Итоговая последовательность развития

```text
Этап 1
Базовый сбор информации об ОС
        ↓
Этап 2
Проверка базовой настройки ВМ
        ↓
Этап 3
Первый профиль СПО: Nginx
        ↓
Этап 4
Поддержка остальных типов СПО и назначение профилей
        ↓
Этап 5
Репликация, синхронизация и актуальность данных
        ↓
Этап 6
Версии прикладного программного обеспечения
```

# Общий критерий завершения этапа

Каждый этап считается завершённым, если:

* [ ] Реализован заявленный сбор данных.
* [ ] Реализована локальная оценка результата, если она предусмотрена этапом.
* [ ] Ошибка сбора отличается от отрицательного результата проверки.
* [ ] Метрики доступны через `node_exporter`.
* [ ] Проверена корректность значений метрик.
* [ ] Проверено влияние приложения на ресурсы ВМ.
* [ ] Документирован формат конфигурации этапа.
* [ ] Документирован способ проверки результата.
